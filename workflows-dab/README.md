# workflows-dab — Laktory + DAB integration path

This directory shows how to integrate Laktory into an existing Databricks Asset Bundles (DAB) project. Laktory pipelines are declared as YAML files; a Python hook converts them into native DAB resources at bundle resolution time, so everything deploys through the standard `databricks bundle` CLI.

For teams using Laktory without DAB, see [`../workflows/`](../workflows/README.md) instead.

## How Laktory integrates with DAB

The `databricks.yml` declares a Python resources hook:

```yaml
python:
  venv_path: .venv
  resources:
    - 'laktory.dab:build_resources'
```

When you run `databricks bundle deploy`, the CLI calls `laktory.dab:build_resources`, which:

1. Reads all pipeline YAMLs from `laktory/pipelines/`
2. Generates DLT pipeline or Job DAB resources (registered under keys like `dlt-pipeline-{name}`)
3. Writes supporting files (pipeline config JSON, DLT notebook) to `laktory/.build/`, which is synced to the workspace

Pipelines defined this way are fully interoperable with hand-written DAB resources. A job YAML in `resources/` can reference a Laktory-generated pipeline with `${resources.pipelines.dlt-pipeline-{name}.id}`.

## Directory structure

```
databricks.yml              DAB bundle config — targets, variables, Laktory hook
laktory/
  pipelines/                Laktory pipeline YAML definitions (your source of truth)
    pl-stock-prices.yml
    pl-taxi-trips.yml
  .build/                   Auto-generated by Laktory — do not edit, committed to .gitignore
resources/                  Hand-written DAB resource YAMLs (jobs, etc.)
  stock-prices.job.yml
  taxi-trips.job.yml
src/
  lake/                     Custom Python package (wheel) deployed alongside pipelines
    dataframe_ext.py        Narwhals namespace extensions callable from pipeline transformers
tests/                      pytest suite using Databricks Connect
fixtures/                   JSON/CSV test data for local tests
```

## What's deployed

| Resource | Type | Description |
|---|---|---|
| `pl-taxi-trips` | Laktory pipeline → DAB Job | Intro pipeline from `samples.nyctaxi.trips` |
| `pl-stock-prices` | Laktory pipeline → DAB DLT Pipeline | Full medallion pipeline with streaming, expectations, quarantine, quality monitor |
| `job-stock-prices` | DAB Job | Daily job wrapping the stock prices DLT pipeline |
| `job-taxi-trip` | DAB Job | Daily job wrapping the taxi trips pipeline |

## Getting started

**1. Install dependencies**
```bash
uv sync --dev
```
This sets up the `.venv` that `laktory.dab:build_resources` runs in during bundle resolution.

**2. Authenticate**
```bash
databricks configure
```

**3. Deploy to dev**
```bash
databricks bundle deploy --target dev
```
`dev` is the default target. Resources are deployed in development mode (schedules paused, resources prefixed with your username).

**4. Deploy to production**
```bash
databricks bundle deploy --target prd
```

**5. Run a resource**
```bash
databricks bundle run job-stock-prices
```

**6. Run tests locally**
```bash
uv run pytest
```
Tests use Databricks Connect. If no compute is configured, serverless compute is used automatically.

## Adding a new pipeline

1. Create a YAML file in `laktory/pipelines/` following the existing examples
2. Optionally create a job in `resources/{name}.job.yml` that references the generated pipeline:
   ```yaml
   pipeline_id: ${resources.pipelines.dlt-pipeline-{name}.id}
   ```
3. Run `databricks bundle deploy --target dev`

The DAB resource key is `dlt-pipeline-{pipeline-name}` by default for DLT pipelines. You can override it by setting `resource_name` in the pipeline YAML's `orchestrator` block:

```yaml
orchestrator:
  type: DATABRICKS_PIPELINE
  resource_name: pl-stock-prices   # DAB key becomes pl-stock-prices instead of dlt-pipeline-pl-stock-prices
```

`pl-stock-prices` uses this override, so its job references `${resources.pipelines.pl-stock-prices.id}`. `pl-taxi-trips` does not, so its job references `${resources.pipelines.dlt-pipeline-pl-taxi-trips.id}`.
